👤 Leads — Prospective Customers
Classes 1 & 2 · Sales Process · Lead Conversion
📚Key theory from the course⌄

What is a Lead?

  • A Lead is a prospective customer. It can be created manually, via Web-to-Lead, LinkedIn Lead Gen, or other sources.
  • It follows a Lead Process with stages defined in the Status picklist field.
  • Once qualified, it is converted into an Account + Contact + (optional) Opportunity.
  • Standard fields are mapped automatically on conversion. Custom fields require manual mapping in Setup → Lead Fields → Map Lead Fields.
  • You can merge up to 3 duplicate leads. Only the lead owner, their manager, or the system admin can do this.

⚡ Lead Automation
Class 2 · Web-to-Lead · Assignment Rules · Lead Queue · Auto-Response
📚Key theory from the course⌄

Lead automation tools

  • Web-to-Lead: generates HTML for a web form. Limit: 500 leads/day. Standard and custom fields can be included. Automatically triggers Assignment Rules and Auto-Response Rules.
  • Lead Assignment Rules: only one active rule at a time. Each rule has multiple entries evaluated in order. No match → default owner set in Lead Settings.
  • Lead Queue: when a queue is created, a list view is automatically created too. Members can be individual users, roles, or public groups.
  • Auto-Response Rules: send an automatic email to the lead when it is created. Only one active rule. The email template is selected per rule entry criterion.
  • Lead Process: a sequence of Status field values. A Lead object can have multiple processes, each assigned to a Record Type.
💡
Golden rule for Assignment Rules: only one rule can be active at a time, but that rule can have as many rule entries as needed. Each entry has a criterion and an action (assign to user or queue).

🏢 Accounts & Contacts
Classes 1 & 4 · Record Types · Duplicate Rules · Validation Rules
📚Key theory from the course⌄

Accounts as the backbone of Salesforce

  • The Account is the backbone of Salesforce: related to Contacts, Opportunities, Cases, and Activities.
  • Record Types let the same object have different fields, layouts, and processes for Individual vs Business accounts — without duplicating the object.
  • Contacts represent individual people, always related to an Account (1 account → N contacts).
  • Duplicate Rules surface potential duplicates when creating or editing. Account merging requires the owner, their manager, or the admin.
  • Validation Rules prevent saving incorrect data. They are evaluated before the record is saved.
Remember (Class 7): Record Types can have different Business Processes, Paths, page layouts, and picklist values on the same object. A Record Type is assigned to profiles. Users can VIEW records of any Record Type, but can only CREATE the ones assigned to their profile.

📚Key theory from the course⌄

Opportunity tools

  • The Path is linked to the Stage field and to the Sales Process. It can display up to 5 key fields per stage.

📦 Products & Pricing
Class 4 · Product2 · Pricebooks · Product Schedules · Quotes
📚Key theory from the course⌄

Products and pricing

  • Standard Pricebook: the master list of all products with their standard prices. Unique — only one per org.
  • Custom Pricebook: an alternative list with custom prices (e.g., corporate rates, peak season).
  • Standard Price: the product's price in the Standard Pricebook. List Price: the price in a Custom Pricebook. Sales Price: the price negotiated by the sales rep.
  • Product Schedules: define payment and delivery cycles. Set the number of installments and frequency.
  • Products are added to opportunities, orders, or service contracts as Opportunity Products (Line Items).

📚Key theory from the course⌄

Campaigns

  • Campaigns track marketing initiatives. Members are leads, contacts, or person accounts.
  • Default Member Status values are Sent and Responded. Custom statuses can be added per campaign. Only one can be the default.

📚Key theory from the course⌄

Important Case fields

  • Case Number: auto-number, read-only, assigned automatically on creation.

⚙ Case Automation
Class 4 · Email-to-Case · Assignment · Escalation · Auto-Response · Web-to-Case
📚Key theory from the course⌄

Case automation tools

  • Email-to-Case: two variants. Standard: requires an on-premise agent, supports attachments over 25 MB. On-Demand: no installation needed, attachments under 25 MB. Once enabled, Email-to-Case cannot be disabled.
  • Web-to-Case: HTML form embedded on a website. Limit: 5,000 cases/day (much higher than Web-to-Lead).
  • Escalation Rules: one active rule, up to 5 escalation actions per entry. Actions trigger if the case isn't closed after N hours.
  • Case Auto-Response Rules: sent to the contact on the case. Only one email per criterion. One active rule at a time.
  • Case Assignment Rules trigger when a case is created (manually, by email, web, or community).
⚠
Important: Case Auto-Response Rules only send to the case contact or the "sender." You cannot send to multiple recipients with a single entry.

Module 9
📄 Object Manager — Objects & Relationships
Classes 5, 6 & 7 · Standard vs Custom · Lookup vs Master-Detail · Junction Objects
📚Key theory from the course⌄

Objects and relationships

  • Standard Objects cannot be deleted. Custom Objects can (API name ends in __c).
  • Lookup: loose coupling, no Roll-Up, parent not required, no cascade delete. Up to 40 per object.
  • Master-Detail: tight coupling, enables Roll-Up, parent required, cascade delete. Max 2 per object. Parent controls the Owner of the child.
  • Junction Object (many-to-many): an object with 2 Master-Detail relationships. Example: Contact ↔ Class (via Class Registration).
  • Cross-object Formulas: let you reference parent fields from a child formula field. Work with both Lookup and Master-Detail.
💡
Note: although Account→Opportunity looks like a Lookup, Salesforce applies special data integrity rules on standard related objects. It's not a Master-Detail, but it prevents deleting an Account that has active Opportunities.

Module 10
🔗 Fields & Relationships
Class 6 · Field types · Roll-Up · Formula · FLS · Picklist Dependencies
📚Key theory from the course⌄

Field types and key concepts

  • Roll-Up Summary: Master-Detail only. Created on the parent, extracts data from the child. Operations: COUNT, SUM, MIN, MAX. Max 25 per object. Read-only.
  • Formula Field: calculates in real time (does not store the value). Can reference fields on the same record or from a parent (cross-object). Types: Checkbox, Date, Number, Text, Currency, Percent.
  • Field-Level Security (FLS): Visible = editable, Read-Only = view only, unchecked = hidden. Best practice: manage via Permission Sets rather than profiles.
  • Picklist Dependencies: a controlling field determines what values the dependent field shows. The controlling field must be a picklist or checkbox.
  • Field History Tracking: can be enabled for up to 20 fields per object to track value changes. History appears in the "Field History" related list.

Layouts and record types

👥 Users, Profiles & Roles
Classes 8 & 10 · User Creation · Licenses · Profiles · Permission Sets · OWD · Role Hierarchy
📚Key theory from the course⌄

Layered security model

  • Profiles → Structure: control what the user can do — which objects, fields, layouts, and apps they can access. Library analogy: what you're allowed to do inside the library.
  • Roles → Data: control which records the user can see. Users higher in the hierarchy see records owned by their subordinates. Library analogy: which specific books you can access.
  • OWD (Org-Wide Defaults): the most restrictive access baseline. Private = only the owner (and hierarchy) can see. Subsequent layers can only open access, never restrict it.
  • Sharing Rules: extend access horizontally (between roles or groups at the same level).
  • Permission Sets: can only add permissions, never remove them. A user can have multiple Permission Sets.
  • Only in profiles: Login Hours, IP Ranges, Page Layout Assignment, default Record Type, default App.
Classroom analogy (library): Profiles define what you can do inside the library (read, write, access sections). Roles determine which specific books or information you can access.

Module 13
💾 Data Management
Classes 10, 11 & 12 · Import Wizard · Data Loader · Mass Transfer · External ID · Storage
📚Key theory from the course⌄

Data management tools

  • Import Wizard: web-based, up to 50,000 records, Insert/Update only, no Delete or Export.
  • Data Loader: installable client, up to 5,000,000 records, all operations including Hard Delete and Export All.
  • Upsert: if record exists → UPDATE, if not → INSERT. The go-to operation for integrations. Works with Record ID or External ID.
  • External ID: a field marked as an external identifier (Text/Number/Email type). Used to relate records with external systems. Max 25 per object.
  • Record ID: 15 or 18 characters. The 18-char version is case-insensitive and recommended for APIs. Never changes — even if a record is deleted and restored.
  • Mass Transfer: changes record ownership in bulk. Does not move data between objects. Requires being above in the hierarchy or having Modify All.
  • Enterprise Edition storage: minimum 10 GB data + 10 GB files. +20 MB data and +2 GB files per licensed user.
🚫
Critical warning: during any data load operation, Validation Rules, Flows, Workflow Rules, and Triggers run. A mass load can trigger thousands of automations simultaneously. Always test in Sandbox first.
